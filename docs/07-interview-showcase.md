# 面试项目展示指南

本文档汇总所有模块的面试要点，帮你在面试前快速复习。包括：项目一句话介绍、架构讲解思路、高频技术问答、常见追问应对。

## 1. 项目一句话介绍

> "我独立设计开发了一个基于 Python + FastAPI + LLM 的求职 AI 助手平台，实现了 RAG 知识库问答、AI 模拟面试、简历智能分析等功能，使用 Celery 处理异步任务，Docker Compose 一键部署。"

**关键词密度：** Python、FastAPI、LLM、RAG、模拟面试、异步任务、Docker — 每个都是 AI 应用岗位的加分项。

## 2. 项目架构讲解（2 分钟版）

按"总 → 分 → 亮点"的结构讲：

### 总览

"这个项目分为三层：
1. **API 层**：FastAPI 提供 RESTful API + WebSocket
2. **业务层**：Orchestrator 统一调度，5 个功能模块
3. **基础设施层**：PostgreSQL 持久化、Redis 缓存/队列、ChromaDB 向量检索"

### 核心模块

"5 个模块按优先级：
1. **RAG 知识库** — 文档上传 → 分块 → 向量化 → 检索增强问答 → SSE 流式输出
2. **模拟面试** — Skill 驱动出题 → 多轮追问 → 分批评估 → 报告生成
3. **简历分析** — 多格式解析 → Celery 异步处理 → 内容哈希去重 → PDF 报告导出
4. **语音面试** — WebSocket 全双工 → ASR 识别 → 复用面试引擎 → TTS 播放
5. **JD 分析/简历优化** — 最初做的基础功能，Tool Calling + Trace 记录"

### 亮点收束

"我觉得这个项目有几个设计上的亮点：
- Trace 记录贯穿全链路，每次请求都能看到完整的执行轨迹
- 模拟面试和语音面试共用同一套评估引擎，I/O 和逻辑分离
- 渐进式架构——先 SQLite 后 PostgreSQL、先 ChromaDB 后 pgvector"

## 3. 高频技术问答

### 架构设计类

**Q: 你这个项目的整体架构是什么？**
> 三层架构。API 层用 FastAPI，支持 REST + WebSocket + SSE 三种通信模式。业务层有 Orchestrator 统一调度，负责任务分发、Trace 记录、错误处理。基础设施层包括 PostgreSQL（持久化）、Redis（缓存 + 队列）、ChromaDB（向量检索）。

**Q: 为什么选 FastAPI？**
> 三个原因：(1) 异步原生，基于 ASGI，适合 LLM 流式调用和 WebSocket；(2) 自动 API 文档（Swagger UI），开发效率高；(3) 与 Pydantic 深度集成，请求/响应自动校验。

**Q: 你的 Orchestrator 模式是怎么设计的？**
> 所有任务通过统一入口 POST /task 进入，Orchestrator 负责：(1) 任务识别——验证 task_type 是否合法；(2) Prompt 加载——从 Markdown 文件读取对应的提示词；(3) LLM 调用——支持普通文本和 Tool Calling 两种模式；(4) 结果汇总——如果有工具调用，执行工具后二次调用 LLM。每一步都记录 Trace 事件，包含节点名、状态、时间戳和备注。

### RAG 类

**Q: 你的 RAG 系统流程是什么？**
> 文档上传 → PyMuPDF/python-docx 提取文本 → RecursiveCharacterTextSplitter 分块（500字，重叠100字）→ OpenAI Embedding 向量化 → 存入 ChromaDB。查询时，问题向量化后做 top-5 相似度搜索，检索到的文档块拼入 prompt，LLM 基于真实文档生成回答。通过 SSE 流式返回。

**Q: 分块参数怎么选的？**
> chunk_size=500 是经验平衡值——太小会丢上下文，太大引入噪声。overlap=100 保证分块边界处信息不丢失。分割器优先按段落、句子切割，尽量不在语义中间断开。实际可通过评估检索准确率来调优。

**Q: RAG 效果不好怎么优化？**
> 四个方向：(1) 分块策略——根据文档类型选择不同的 splitter；(2) 混合检索——向量搜索 + BM25 关键词匹配；(3) Re-ranking——用交叉编码器对初步检索结果重排序；(4) Query Expansion——用 LLM 改写问题提高检索命中率。

### 异步处理类

**Q: 简历分析为什么要异步？**
> LLM 分析耗时 5-30 秒，同步会让用户等待且 HTTP 可能超时。用 Celery + Redis 实现异步：用户上传后立即返回"分析中"，Celery Worker 后台处理完更新数据库状态。支持最多 3 次重试。

**Q: Celery 和 asyncio 的区别？什么时候用哪个？**
> asyncio 适合 I/O 密集的并发场景（如同时调多个 API），运行在同一进程中。Celery 是分布式任务队列，任务运行在独立的 Worker 进程中，支持持久化、重试、监控、定时任务。简历分析这类重任务适合 Celery——Worker 崩溃重启后任务不丢失。

### 模拟面试类

**Q: 多轮对话上下文怎么管理？**
> Redis 存储面试 session，包含完整 messages 数组。每轮回答后更新 session。设置 2 小时 TTL 自动过期。面试通常 10-20 轮，在 token 限制内用全量历史。更长的可以切到滑动窗口 + 摘要压缩。

**Q: 评估引擎怎么设计的？**
> 分批评估架构：面试结束后，按每 3 题一批调用 LLM 评分（Structured Output 输出分数和评语），然后二次汇总生成总评报告（总分、亮点、改进建议）。分批的好处是 LLM 注意力更集中、可并发加速。

### 部署类

**Q: 项目怎么部署？**
> Docker Compose 一键部署 4 个服务：FastAPI API、Celery Worker、PostgreSQL、Redis。服务间通过 Docker 网络通信。healthcheck 确保启动顺序。Volume 挂载持久化数据。

## 4. 常见追问应对

### "你在项目中遇到过什么难点？"

准备 2-3 个真实的技术问题：

1. **RAG 检索质量不稳定**
   > "初期检索经常返回不相关的内容。排查发现是分块太大导致向量不够精确。把 chunk_size 从 1000 调到 500，并加了 100 字重叠，检索准确率明显提升。后续还计划加混合检索。"

2. **Celery 任务重试后数据不一致**
   > "简历分析任务重试时，如果第一次已经写了部分结果到数据库，重试会重复写入。解决方案是加了幂等性检查——用内容哈希判断是否已分析过，如果有直接跳过。"

3. **模拟面试评估引擎评分不一致**
   > "发现同一份回答多次评估分数差异大。改进方案：(1) 将 temperature 设为 0 减少随机性；(2) 用 Structured Output 强制 JSON 格式，避免解析错误；(3) 在评分 prompt 中给出明确的评分标准和示例。"

### "你觉得项目有什么不足？"

展示自我认知和改进方向：

> "几个方面：(1) 前端比较简陋，用的 Streamlit，后续可以换 React 做更好的交互；(2) 向量检索只用了相似度搜索，还没加 Re-ranking 和混合检索；(3) 语音面试延迟还比较高，没有做客户端 VAD 和 WebRTC；(4) 缺少完善的监控和日志收集。"

### "如果让你重新做，你会怎么改？"

> "两个方面：(1) 数据库从一开始就用 PostgreSQL + pgvector，避免后期迁移的工作量；(2) 前端一开始就用 React，Streamlit 虽然快但扩展性不好。但在 1-3 个月的时间约束下，先用简单方案快速跑通、再逐步优化是合理的工程取舍。"
