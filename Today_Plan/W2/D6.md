## 编码任务

### 任务 1：理解当前单元测试（Doc 02 §5）

当前 `tests/test_rag_chain.py` 已存在，今天的重点不是从零创建，而是读懂它现在到底在验证什么。

**当前真实覆盖**：

| 测试函数 | 测试内容 | 断言 |
|---|---|---|
| `test_rag_query_returns_fallback_when_no_docs` | 检索为空时直接走 fallback | 返回固定 `answer` + 空 `sources` |
| `test_rag_query_returns_answer_and_sources_when_docs_found` | 命中检索结果时拼接 answer + sources | answer 正确；sources 保留 `source_file` 和 `chunk_index` |
| `test_rag_query_truncates_source_content_to_50_chars` | source 预览内容截断 | `content` 长度为 50 |
| `test_yields_fallback_when_no_docs` | 流式路径无检索结果 | 只产出一次 fallback |
| `test_yields_chunks_when_docs_found` | 流式路径命中结果 | 按底层 `astream` 顺序透传 chunk |

**学习重点**：
- 理解 mock 的作用：用假 `search()` / 假 chain 隔离真实向量库和 LLM
- 理解这些测试测的是 `rag_query()` / `rag_query_stream()` 的分支逻辑，而不是底层向量检索质量
- 能说清楚“无命中时为什么直接 fallback，而不是继续调用 LLM”

### 任务 2：理解当前集成测试

对齐当前 `tests/test_kb_api.py` 的真实覆盖范围，重点理解它如何测试 HTTP 契约，而不是把它当成“待补齐的最小草稿”。

**当前真实覆盖重点**：

| 测试方向 | 当前关注点 |
|---|---|
| upload 成功 | HTTP 响应、`KnowledgeDocument` 记录（含 `updated_at`）、chunks_count 同步成立 |
| upload 失败 | 不支持格式返回 400；向量写入失败时保留 `status=failed` 记录、补偿删除向量、清理落盘文件 |
| upload 幂等短路 | 同 hash 二次上传返回 `reused: true`，不调用 `load_and_split` / `add_documents` |
| upload 并发冲突 | `IntegrityError` → HTTP 409 |
| 近重复确认 | `confirmation_required`、`confirm_upload=true` 重试、跨 collection 不触发、降级放行 |
| query | `/kb/query` 使用 JSON body，返回 `answer + sources` |
| collections | 直接反映 Chroma 当前 collection 状态 |
| stream | SSE 协议只承诺 `message/done` 两类事件 |

**学习重点**：
- 用测试专用 `FastAPI()` 挂载知识库 router，不直接复用 `app.main.app`，避免触发真实 lifespan
- 用 `dependency_overrides[get_db]` 把请求期 session 指到测试库
- `test_kb_api.py` 以 monkeypatch 为主，隔离真实向量库、模型与主应用启动副作用
- `/kb/query` 和 `/kb/query/stream` 都按 JSON body 发请求
- 理解 `reused=True` 的含义：请求成功，但方式是复用旧结果而不是重新处理文件
