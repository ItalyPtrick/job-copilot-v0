## 学习内容（~1h）

复习 Doc 04 §6 面试要点，确保能流畅回答：

### 必须能答的 4 个问题

1. **Q: 简历分析为什么要异步处理？**
   > LLM 调用 5-30 秒，同步会让用户等待且 HTTP 可能超时。异步让用户立即得到反馈，后台 Worker 完成后更新状态。

2. **Q: Celery 重试机制怎么工作？**
   > `max_retries=3, default_retry_delay=10`。Task 中 `raise self.retry(exc=e)` 触发重试。3 次都失败标记为 FAILURE。

3. **Q: 内容哈希去重的作用？**
   > SHA-256 对纯文本做哈希，数据库加唯一索引。相同简历直接复用已有结果，节省 LLM 成本。

4. **Q: PDF 中文支持怎么做？**
   > ReportLab 注册 TTF 中文字体，样式中指定字体名。开发用系统字体，生产打包开源字体。

### 5 个能讲的亮点

1. 异步处理管道（Celery + Redis）
2. 自动重试（LLM 失败不丢任务）
3. 内容哈希去重（省钱省时间）
4. 多格式统一解析（策略模式）
5. PDF 报告导出（可下载的成果物）

  SHA-256 on raw text，数据库唯一索引

  开发用系统字体 fallback，生产打包开源字体

### 任务 6：模块复盘（AI 面试官拷打）

AI 切换为面试官角色，向用户提 3-5 个技术问题，参考方向：

1. **架构设计**："你的简历分析模块从上传到出报告，数据流经过哪几个阶段？每个阶段的职责是什么？"
2. **异常处理**："如果 LLM 返回的 JSON 格式错误，你的系统怎么处理？如果 3 次重试都失败呢？"
3. **性能优化**："为什么用 Celery 而不是 asyncio.create_task？如果同时有 100 人上传简历会怎样？"
4. **去重策略**："内容哈希去重的边界在哪？同一份简历换目标岗位会怎样？"
5. **可扩展性**："如果要新增支持 Markdown 格式的简历，需要改哪里？改动大吗？"

用户口头回答，AI 追问 + 给出评价。
