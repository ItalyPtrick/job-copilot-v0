## 学习内容（~45 分钟）

理解今天的设计变化：

1. **Skill 不再只是每次全文塞进 prompt**
   - Markdown Skill 仍是配置源。
   - 应用层先把 Skill 转成结构化“面试蓝图”：考察范围、难度分布、参考知识库、难度语义。
   - 每次出题只传当前需要的蓝图摘要、已覆盖维度、已问列表和候选人上下文，降低重复 token 和注意力稀释。

2. **难度控制不只传 easy / medium / hard 标签**
   - `easy`：基础概念、定义、简单辨析。
   - `medium`：原理解释、场景应用、常见坑。
   - `hard`：系统设计、权衡、故障分析、性能与边界。
   - LLM 返回题目时必须说明 `difficulty_reason`，应用层校验返回难度与请求一致。

3. **追问函数只负责生成追问，不负责决定是否追问**
   - 是否追问由 D5 的 planner 决定。
   - D3 的 `generate_follow_up` 只接收原题、回答、追问焦点，生成一句自然追问。

## 编码任务

### 任务 1：扩展 Skill 出题数据结构

在 `app/modules/interview/schemas.py` 或 `question_engine.py` 中补充当前阶段需要的轻量结构：

- `SkillBlueprint`：从 Markdown Skill 提取出的面试蓝图。
- `DifficultyRubric`：`easy / medium / hard` 的语义说明。
- `InterviewQuestion` 增加或预留：`difficulty_reason`、`assessment_focus`。

> 如果不想马上改 schema，可先在 `question_engine.py` 内部用 dict 表示，但测试要固定返回契约。

### 任务 2：重构 Skill 加载与蓝图构建

`question_engine.py` 中保留：

- `load_skill(skill_name)`：读取 `app/skills/{skill_name}.md`。
- `build_skill_blueprint(skill_content)`：把 Markdown 转成结构化蓝图；当前可先用规则解析，后续再考虑 LLM 解析。

蓝图至少包含：

```python
{
    "topics": [...],
    "difficulty_distribution": {"easy": 0.4, "medium": 0.4, "hard": 0.2},
    "reference_collections": [...],
    "difficulty_rubric": {...},
}
```

### 任务 3：重构 `generate_question`

`generate_question` 改为接收蓝图摘要，而不是每次直接传整份 Markdown：

- 输入：`skill_blueprint`、`difficulty`、`asked_questions`、`covered_topics`、`candidate_context`。
- prompt 明确包含：当前目标难度的 rubric、已问问题、已覆盖考点、这次希望覆盖的考点。
- 返回：`question / category / difficulty / difficulty_reason / follow_up_hint / assessment_focus`。
- 应用层校验：
  - 请求难度只能是 `easy|medium|hard`。
  - 返回难度必须等于请求难度。
  - 返回题目不能和 `asked_questions` 重复。

### 任务 4：保留并收紧 `generate_follow_up`

`generate_follow_up` 的职责固定为“已决定追问时生成追问文本”：

- 输入：原始问题、候选人回答、追问焦点、可选最近上下文。
- 输出：一句追问问题文本。
- 继续兼容 `call_llm` 返回纯文本时的 `raw` 字段。
