## 学习内容（~30 分钟）

快速复习 Doc 03 §1 "Skill 驱动出题"和 §4 Step 3 的代码，理清两个核心函数的调用链：

1. **`generate_question`**：读取 Skill 文件 → 构造 system_prompt（含考察范围 + 难度 + 已问题目列表）→ 调用 `call_llm` → 返回结构化 JSON（question/category/difficulty/follow_up_hint）
2. **`generate_follow_up`**：接收原始问题 + 候选人回答 + 追问方向提示 → 构造追问 prompt → 调用 `call_llm` → 返回追问文本

关键设计决策：
- **去重机制**：把 `questions_asked` 列表传入 prompt，要求 LLM 不重复；同时在应用层维护已问列表
- **难度控制**：通过 difficulty 参数指定，后续路由层按 `difficulty_distribution` 分配

## 编码任务

### 任务 1：创建出题引擎（Doc 03 Step 3）

创建 `app/modules/interview/question_engine.py`，参考 Doc 03 Step 3 的代码：

核心内容：
- `generate_question(skill_content, difficulty, asked_questions, context="")` → dict
  - 构造 system_prompt：角色设定 + Skill 定义 + 难度级别 + 已问题目列表 + JSON 输出格式要求
  - 调用 `call_llm(system_prompt, {"context": context})`
  - 返回值应包含 `{"question", "category", "difficulty", "follow_up_hint"}`

- `generate_follow_up(original_question, candidate_answer, follow_up_hint)` → str
  - 构造追问 prompt：原始问题 + 候选人回答 + 追问方向
  - 调用 `call_llm(system_prompt, {})`
  - 返回追问的问题文本

**依赖**：
- `from app.services.llm_service import call_llm` — 项目已有的 LLM 调用封装

**注意（⚠️ 与现有 `call_llm` 的兼容问题）**：
- `call_llm` **始终返回 dict**——即使 LLM 返回纯文本，也会因 JSON 解析失败而返回 `{"error": "模型返回格式异常", "raw": "实际文本内容"}`
- `generate_follow_up` 的 prompt 要求 LLM "只返回追问的问题文本"，LLM 大概率返回纯文本 → `call_llm` 走兜底分支
- **正确的兜底写法**：`return result.get("raw") or result.get("question") or str(result)`
  - 优先取 `raw`（纯文本被 call_llm 放在这个字段里）
  - 其次取 `question`（如果 LLM 恰好返回了 JSON 带 question 字段）
  - 最后 `str(result)` 兜底
- ❌ Doc 03 原文的 `result.get("question", str(result))` 会跳过 `raw` 字段，导致返回整个 dict 的字符串化

### 任务 2：创建 Skill 加载辅助函数

在 `question_engine.py` 中（或单独文件）添加 Skill 文件加载函数：

```python
def load_skill(skill_name: str) -> str:
    """加载 Skill Markdown 文件内容"""
    skill_path = f"app/skills/{skill_name}.md"
    with open(skill_path, "r", encoding="utf-8") as f:
        return f.read()
```
