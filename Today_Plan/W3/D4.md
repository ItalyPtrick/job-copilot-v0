## 学习内容（~45 分钟）

新版追问策略会让一条主问题下出现 0~N 条追问，因此评估不能再简单使用“assistant 含问号 + 下一条 user”作为独立题目。今天要理解新的评估单元：

```text
主问题
  ├─ 候选人第一次回答
  ├─ 可选追问 1
  ├─ 候选人追问回答 1
  └─ 可选追问 2 ...
```

评估时应以“主问题轮次”为单位：主问题考察什么、候选人初答质量如何、追问暴露了什么、最终掌握程度如何。

## 编码任务

### 任务 0：固定 interview message contract

从 D4 开始，评估和路由共享同一套消息结构：

```python
{
    "role": "assistant",
    "content": "...",
    "metadata": {
        "question_type": "main",
        "question_id": "q_1",
        "parent_question_id": None,
        "category": "Python 基础",
        "difficulty": "easy",
        "assessment_focus": "考察 GIL 的概念和影响",
    },
}
```

约束：
- 主问题：`question_type="main"`，`parent_question_id=None`
- 追问：`question_type="follow_up"`，`parent_question_id` 指向主问题 ID
- 用户回答：metadata 至少包含 `answer_to_question_id`
- `current_question_index` 只统计主问题，不统计追问

### 任务 1：创建面试轮次提取函数

创建 `app/modules/interview/evaluation.py`，实现：

- `_extract_interview_turns(messages)`
  - 从 session messages 中提取结构化轮次。
  - 区分 `question_type=main` 与 `question_type=follow_up`。
  - 每个 turn 包含主问题、初答、追问列表、考察方向、难度。

D5 路由必须按上面的 message contract 写入 session；D4 测试先用该 contract 构造 fixture。

### 任务 2：实现 `evaluate_batch(turns)`

- 每批处理 2-3 个主问题轮次，而不是 3 个裸问答对。
- prompt 包含评分标准：
  - 1-3：概念错误或答非所问
  - 4-5：知道概念但缺少原理
  - 6-7：能解释原理并处理常见场景
  - 8-9：能结合经验、边界和权衡
  - 10：回答完整且有结构化思考
- 单题评分要综合主问题回答和追问回答。

### 任务 3：实现 `generate_report(all_evaluations)`

返回：

- `overall_score`
- `summary`
- `strengths`
- `improvements`
- `items`

报告需要突出：候选人的强项、薄弱考点、追问后是否能补救。

### 任务 4：实现 `evaluate_interview(messages)`

流程：

```text
messages → _extract_interview_turns → evaluate_batch 分批 → generate_report 汇总
```
