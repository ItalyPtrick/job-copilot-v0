## 学习内容（~1.5 小时）

阅读 Doc 03 §1 的前 2 个小节：

### 1. 多轮对话管理（Doc 03 §1 第 1 节）

- 理解面试对话的核心问题：上下文管理、上下文长度控制、会话状态
- 掌握 4 种上下文策略的区别：

| 策略 | 描述 | 适用场景 |
|---|---|---|
| 全量历史 | 每次把所有对话都发给 LLM | 短对话（<10轮） |
| 滑动窗口 | 只保留最近 N 轮对话 | 长对话，防止超 token |
| 摘要压缩 | 定期用 LLM 压缩历史为摘要 | 超长对话 |
| 混合策略 | 最近 N 轮全量 + 更早的摘要 | 推荐方案 |

- **本项目选择**：全量历史（面试通常 10-20 轮，在 token 限制内），配合 Redis 缓存
- **面试怎么讲**："选全量历史因为面试通常不超 20 轮，token 足够；如果面试更长，可以切换到滑动窗口 + 摘要压缩"

### 2. 结构化输出（Doc 03 §1 第 2 节）

- 理解为什么需要结构化输出：评估引擎需要固定格式的评分 JSON，不能用自由文本
- 掌握 OpenAI `response_format` + Pydantic 的用法：

```python
# 关键代码模式
response = client.beta.chat.completions.parse(
    model="gpt-4o-mini",
    messages=[...],
    response_format=InterviewQuestion,  # Pydantic 类
)
question = response.choices[0].message.parsed  # 直接得到 Pydantic 对象
```

- **面试怎么讲**："用 Pydantic 定义输出 schema，通过 OpenAI structured output 强制 LLM 返回合规 JSON，解析零成本"

重点理解：
- **全量历史 + Redis 缓存** 的组合：Redis 存 session，每次请求取出完整 messages，发给 LLM，再存回去
- **Pydantic 在面试模块的双重角色**：既做 API 请求/响应校验（FastAPI），又做 LLM 输出格式约束

- `InterviewStatus` 枚举：`created / in_progress / completed / evaluating / evaluated`
